dns-edge 技术设计文档

基于 Go + miekg/dns 的高性能权威 DNS 服务，支持 HTTP 热更新 API 与流量分流 Go 1.25 PostgreSQL Nacos

1. 背景与目标

背景

传统权威 DNS（BIND、NSD）的记录变更需要修改 Zone 文件并重新加载，操作繁琐且有短暂中断风险。在微服务、容器化场景下，后端 IP 频繁变更，需要一套能够实时、无重启地更新 DNS 记录的权威 DNS 服务。

同时，随着多 IDC、多可用区部署的普及，同一域名往往对应多个后端，需要根据后端的实时负载或健康状况动态调整流量分配比例。

目标

  • 实现符合 RFC 标准的权威 DNS 服务
  • 提供 HTTP REST API，支持 DNS 记录实时热更新（无需重启）
  • 支持基于权重的流量分流，权重可由外部采样系统动态写入 Nacos DataID
  • 支持多实例水平扩展，实例间保持最终一致
  • 单二进制部署，容器化友好（镜像 < 20MB）

1.3 项目定位

dns-edge 是一个 miekg/dns 项目，不是 CoreDNS 项目。
项目说明
miekg/dnsGo DNS 协议库，提供 RR 类型、ServeMux、PacketConn 等基础设施
CoreDNS基于 miekg/dns 的插件化 DNS 框架，引入 plugin chain / coremain / Corefile 解析器
dns-edge直接使用 miekg/dns，不引入 CoreDNS plugin chain 和 coremain。仅可选借鉴 coredns/caddy 解析 Corefile 风格配置

dns-edge 的配置格式参考 Corefile 块级语法（可读性好），但底层直接调用 miekg/dns API 启动 DNS 服务，与 CoreDNS 框架无关。

3. ZoneStore（内存存储层）

ZoneStore 是所有 DNS 查询的直接数据源，需保证高并发读性能。

数据结构

type ZoneStore struct {
    mu    sync.RWMutex
    zones map[string]*Zone   // key: 域名（FQDN，带尾点）
}

type Zone struct {
    Name    string
    Records map[RecordKey][]*Record  // key: (name, type)
}

type Record struct {
    Name    string
    Type    uint16
    TTL     uint32
    Value   string
    Weight  int   // 0 表示不参与分流
}

并发策略

操作锁策略说明
DNS 查询（读）RLock()允许并发读，零竞争
热更新 API（写）Lock()独占写，低频操作
PG 同步（写）Lock()每 30s 一次，影响极小
第一阶段：RWMutex（默认）。DNS 场景读多写少，RWMutex 读锁本质是原子操作，高并发读下竞争极低，开发期足够使用。

可选升级：atomic.Value COW（零锁读）

压测发现 ZoneStore 锁竞争明显时，升级为 Copy-on-Write 模式，读路径完全无锁：

// 读路径：一次原子 load，零锁
zones := s.snapshot.Load().(map[string]*Zone)

// 写路径：复制旧 map → 修改 → 原子替换
// （写操作本身加 Mutex 防并发写，写是低频操作）
s.mu.Lock()
defer s.mu.Unlock()
old := s.snapshot.Load().(map[string]*Zone)
next := make(map[string]*Zone, len(old))
for k, v := range old { next[k] = v }
next[zone.Name] = zone
s.snapshot.Store(next)
不建议使用 xsync.Map 等第三方并发 Map。其价值在于高并发写不同 key 的场景（如计数器）；DNS ZoneStore 写入整个 Zone 对象且极低频，无额外收益，反而引入外部依赖。

4. DNS QueryHandler

基于 miekg/dns 实现 dns.Handler 接口：

func (h *Handler) ServeDNS(w dns.ResponseWriter, r *dns.Msg) {
    q := r.Question[0]

    // 1. 查找记录
    records := h.store.Lookup(q.Name, q.Qtype)
    if records == nil {
        // NXDOMAIN
        m := new(dns.Msg)
        m.SetRcode(r, dns.RcodeNameError)
        w.WriteMsg(m)
        return
    }

    // 2. 分流选择（如有多后端）
    selected := h.selectByWeight(records)

    // 3. 构造响应
    m := new(dns.Msg)
    m.SetReply(r)
    m.Authoritative = true
    for _, rr := range selected {
        m.Answer = append(m.Answer, rr)
    }
    w.WriteMsg(m)

    // 4. 概率触发 PG 同步
    if rand.Float64() < h.syncProb {
        go h.syncer.TriggerSync()
    }
}

支持的记录类型

A AAAA CNAME MX TXT NS SOA PTR SRV

5. 流量分流

权重来源优先级

Nacos 动态权重  >  PG 静态权重  >  均等分配（fallback）

Nacos 权重格式

DataID: dns_weights:{fqdn}:{type}         // 例：dns_weights:api.example.com.:A
Group:  DEFAULT_GROUP（可按环境配置）
Value:  {"1.2.3.4": 70, "5.6.7.8": 30}

启动时 getConfig 全量拉取；ListenConfig 注册回调，Nacos 推送变更后毫秒级生效。
Nacos 不可用时自动降级为 PG 静态权重，PG 也无权重则均等分配。

加权随机算法

func weightedRandom(records []*Record) *Record {
    total := 0
    for _, r := range records {
        total += r.Weight
    }
    n := rand.Intn(total)
    for _, r := range records {
        n -= r.Weight
        if n < 0 {
            return r
        }
    }
    return records[len(records)-1]
}
采样系统与 DNS 服务完全解耦。DNS 层通过 Nacos ListenConfig 回调感知权重变更，采样系统随时可以独立上线/更新算法，不影响 DNS 服务。

7. PostgreSQL 数据模型

-- 域名表
CREATE TABLE domains (
    id         BIGSERIAL PRIMARY KEY,
    name       TEXT NOT NULL UNIQUE,    -- FQDN，带尾点，如 example.com.
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMPTZ             -- 软删除
);

-- 记录表
CREATE TABLE records (
    id         BIGSERIAL PRIMARY KEY,
    domain_id  BIGINT NOT NULL REFERENCES domains(id),
    name       TEXT NOT NULL,           -- FQDN
    type       TEXT NOT NULL,           -- A / AAAA / CNAME / MX ...
    ttl        INTEGER NOT NULL DEFAULT 300,
    value      TEXT NOT NULL,
    weight     INTEGER NOT NULL DEFAULT 100,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMPTZ             -- 软删除
);

CREATE INDEX idx_records_domain_id  ON records(domain_id);
CREATE INDEX idx_records_updated_at ON records(updated_at);
删除操作使用软删除（设置 deleted_at），不能物理删除。否则增量拉取时无法感知已删除的记录，导致其他实例内存中残留。

增量同步查询

SELECT r.*, d.name AS domain_name
FROM records r
JOIN domains d ON r.domain_id = d.id
WHERE r.updated_at > $1
   OR d.updated_at > $1
ORDER BY r.updated_at ASC;

8. 多实例同步

定时同步（30s 间隔）

goroutine 启动 → 记录 last_sync_at
循环：
  sleep(30s)
  SELECT * FROM records WHERE updated_at > last_sync_at
  批量更新内存 ZoneStore
  更新 last_sync_at

概率触发同步（1% 概率）

在 ServeDNS 路径末尾，以 1% 概率异步触发一次增量同步。配合 Token Bucket 速率限制，防止 QPS 高峰期打爆 PG。

一致性窗口

场景最大不一致窗口
API 写入本实例0ms（直接更新内存）
其他实例定时同步≤ 30s
其他实例概率触发≤ 几秒（高 QPS 下）
最终一致性模型与 DNS TTL 机制天然兼容——下游 resolver 本身会缓存记录到 TTL 过期，30s 的同步窗口在 DNS 语义上完全可接受。

9. AXFR Zone Transfer

多实例或主从部署时，slave 节点可通过 AXFR 协议从 master 拉取完整 Zone 数据。miekg/dns 原生支持 AXFR，实现成本低。

  • 收到 AXFR 查询时，从 ZoneStore 序列化完整 Zone，按协议格式分包返回
  • SOA serial 格式：YYYYMMDDnn（Unix 时间戳风格），每次 API 写操作后自增
  • slave 节点对比 serial 决定是否触发 AXFR，减少不必要的传输

8. 设计原则

模块间只依赖接口

所有跨模块调用通过 Go interface 而非具体类型传递。切换底层实现（如 RWMutex → COW）只需改启动时的依赖注入，调用方代码零修改。

核心接口（实现时细化）

// ZoneStore — DNS 查询和热更新的核心存储
// 第一阶段：RWMutexStore；可选升级：COWStore
type ZoneStore interface {
    Lookup(name string, qtype uint16) []*Record
    Update(zone *Zone) error
    Delete(name string) error
    Snapshot() map[string]*Zone   // 供 AXFR 使用
}

// WeightProvider — 分流权重来源
// NacosWeightProvider    → Nacos ListenConfig 推送，毫秒级感知变更（主）
// StaticWeightProvider   → 从 ZoneStore 读静态权重（PG 持久化，降级用）
// CompositeWeightProvider → Nacos 优先，Nacos 不可用时自动降级静态
type WeightProvider interface {
    GetWeights(fqdn string, qtype uint16) map[string]int
}

// Syncer — PG 增量同步，解耦同步策略与存储实现
type Syncer interface {
    TriggerSync() error
    Start(ctx context.Context)
}

依赖方向

cmd/dns-edge (main)
    │  注入具体实现
    ├── dns.Handler      依赖 ZoneStore + WeightProvider
    ├── api.Handler      依赖 ZoneStore + pg.Store
    └── sync.Scheduler   依赖 ZoneStore + pg.Store

各模块不 import 兄弟模块的包，只 import 公共 interface 包。
切换实现 = 改 main 里的注入，业务代码零改动。
接口细节在写代码时细化，设计原则先定：模块间只传接口，不传具体类型。

Corefile 风格配置

参考 CoreDNS 的块级语法，每个模块只读自己的块，workers 字段直接控制 SO_REUSEPORT 的 goroutine 数量：

dns-edge {
    listen   :5300
    workers  0          # SO_REUSEPORT；0=禁用，-1=NumCPU，默认 0
    tcp      true
    edns0    true       # 启用 EDNS0，UDP 上限 4096B

    api {
        listen :8080
    }

    postgres {
        dsn  postgres://user:pass@host:5432/db?sslmode=require
    }

    nacos {
        addr           nacos:8848
        namespace      ""           # 命名空间 ID（默认 public）
        group          DEFAULT_GROUP
        data_id_prefix dns_weights: # DataID 格式：dns_weights:{fqdn}:{type}
    }

    sync {
        interval  30s
        prob      0.01
        ratelimit 100       # 概率触发最大 QPS
    }
}
第一阶段 workers 0（禁用 SO_REUSEPORT），压测发现单 socket 是瓶颈后改为 workers -1（NumCPU），无需改代码。

性能优化策略

按优先级排列，按需逐步启用，不过早优化。

① ZoneStore 读路径零锁（atomic.Value COW）

问题：RWMutex 在写操作时会短暂阻塞所有读者。方案：atomic.Value 保存 Zone Map 快照，读时 load 指针无锁，写时复制 Map 后 atomic store 替换。适用时机：压测发现 ZoneStore 锁竞争明显时升级。

② DNS 消息 Buffer 复用（sync.Pool）

问题：高 QPS 下每次查询分配 dns.Msg 和字节缓冲区，GC 压力大。方案：用 sync.Pool 复用 []byte 缓冲区，减少堆分配和 GC 停顿，降低 P99 延迟抖动。

var bufPool = sync.Pool{
    New: func() any {
        buf := make([]byte, dns.DefaultMsgSize)
        return &buf
    },
}
buf := bufPool.Get().(*[]byte)
defer bufPool.Put(buf)

③ EDNS0 支持（减少 TCP Fallback）

问题：标准 DNS UDP 响应限 512 字节，大记录（MX/TXT）触发客户端 TCP 重试，增加延迟和连接数。方案：响应中声明 EDNS0（RFC 6891），UDP 上限扩展至 4096 字节，大幅减少 TCP fallback。

opt := &dns.OPT{Hdr: dns.RR_Header{Name: ".", Rrtype: dns.TypeOPT}}
opt.SetUDPSize(4096)
m.Extra = append(m.Extra, opt)

④ UDP 多 Socket 并发读（SO_REUSEPORT）

问题：默认单 socket 由单 goroutine 串行读包，是多核瓶颈。方案：SO_REUSEPORT 让多个 socket 同时绑定同一端口，内核将 UDP 包哈希分发到各 socket，多 goroutine 并发读，消除读取串行瓶颈。

对外只有一个 :5300 端口，内核将包分散到多个 socket：

  外部 :5300 UDP
      │
      ├── socket1 → goroutine1 → handler
      ├── socket2 → goroutine2 → handler   ← 内核哈希分发
      └── socket3 → goroutine3 → handler

Worker 数量通过配置项控制，默认 runtime.NumCPU()
lc := net.ListenConfig{
    Control: func(network, address string, c syscall.RawConn) error {
        return c.Control(func(fd uintptr) {
            syscall.SetsockoptInt(int(fd), syscall.SOL_SOCKET,
                syscall.SO_REUSEPORT, 1)
        })
    },
}
// 启动 dns.workers 个 goroutine，各自 ListenPacket(":5300")

适用时机：单机 QPS 超过 10 万后考虑启用，前期用默认单 socket 即可。

优化启用路径

第一阶段（开发期）：RWMutex + 默认单 socket
    ↓  压测 QPS 瓶颈
第二阶段：启用 SO_REUSEPORT + sync.Pool + EDNS0
    ↓  发现 GC / 锁竞争
第三阶段：升级 atomic.Value COW

13. 后续扩展方向

  • 采样系统集成：独立服务探测后端延迟/成功率，按公式计算权重写入 Nacos DataID（dns_weights:{fqdn}:{type}）
  • DNSSEC：由 dnsdist 层处理，权威 DNS 不实现签名
  • Prometheus 指标：查询 QPS、延迟分布、同步延迟、权重快照
  • Web 控制台：可视化管理 DNS 记录和分流权重
  • gRPC API：作为 HTTP API 补充，供内部服务调用
